# Multi-Provider RAG API Upgrade Plan (Target: 8/10)

This plan outlines the steps to upgrade the Multi-Provider RAG API project to a &quot;credible production system&quot; (an 8/10 rating), addressing critical blockers, high-value improvements, and reconciling documentation with implementation. This plan avoids expanding into new features that would push the project to a 9+/10 tier.

## Locked Design Decisions:

1.  **Scope:** Address all Critical and High-value items identified in the project evaluation.
2.  **Authentication:** Implement an **optional, shared API key** via an environment variable (`RAG_API_KEY`). If set, protected endpoints require an `X-API-Key` header.
3.  **Frontend Serving:** Deploy the React frontend as a **separate service** (nginx serving static files) with backend API access via `CORSMiddleware`.
4.  **CI Strategy:** Implement GitHub Actions CI with a **real Qdrant service container** to run `pytest` (including integration tests) and build the frontend.
5.  **Dependency Migrations:** Perform **full migrations** for `pypdf2` → `pypdf`, `google-generativeai` → `google-genai`, and update the Gemini model ID.
6.  **API Path Strategy:** All **business endpoints** (`/ask`, `/compare`, `/evaluate`, `/documents`, `/collections`) will be mounted under an `/api` prefix. `/health`, `/readiness`, and the root `/` index will remain at the root path for infrastructure probes.
7.  **Frontend Scope:** Fix known bugs (`Ask.tsx` non-stream branch, deprecated `onKeyPress`) and ensure it works against the new backend contract (`/api` prefix, error handling). No new feature development.
8.  **`/collections` N+1 Fix:** Implement a **non-invasive approach** using a bounded distinct-`doc_id` scroll with a short TTL cache (`cachetools.TTLCache`) for `list_collections`. This avoids payload schema changes and data migration.

## Implementation Plan (Phased Approach):

### Phase A — Backend Correctness &amp; Foundational Fixes

**Goal:** Resolve core correctness issues and event-loop blocking.

1.  **Fix `/readiness` Endpoint (Critical #1):**
    *   **Description:** The `AppState.qdrant_connected` flag is never set.
    *   **Action:** In `app/main.py`'s `lifespan` event, add a Qdrant client probe (e.g., `client.get_collections()`) within a `try/except` block. If successful, set `state.AppState.qdrant_connected = True`.
    *   **Validation:** `GET /readiness` returns `"status": "ready"` when Qdrant is connected.

2.  **Prevent Unhandled Exception Leakage (Critical #4):**
    *   **Description:** The `unhandled_exception_handler` in `app/errors.py` returns `str(exc)`, which can expose internal details.
    *   **Action:** Modify `app/errors.py:unhandled_exception_handler` to return a generic message like `"An unexpected internal error occurred"` to clients, while logging the full exception server-side.
    *   **Validation:** Trigger a simulated internal error and verify the client receives a generic message without internal details.

3.  **Wire `response_model` on all Routers (High #3):**
    *   **Description:** Pydantic schemas in `app/models/schemas.py` are defined but not used as `response_model` arguments in router definitions, leading to incomplete OpenAPI docs and no output validation.
    *   **Action:** For every endpoint in `app/routers/` that returns a structured response, add `response_model=YourResponseSchema` to the `@router.post` / `@router.get` decorator. Create new response schemas in `app/models/schemas.py` for endpoints like `/compare` that lack them.
    *   **Validation:** Verify FastAPI's `/docs` UI shows correct and complete response schemas for all endpoints.

4.  **Fix Event-Loop Blocking Embedder (High #1):**
    *   **Description:** `EmbeddingService.embed_texts` is `async def` but calls the synchronous `_model.encode()` directly, blocking the event loop.
    *   **Action:** In `app/services/embedder.py`, wrap the `self._model.encode(...)` call within `await asyncio.to_thread(...)`.
    *   **Validation:** Implement a simple load test (e.g., `locust`) hitting `/ask` concurrently and observe event loop utilization (e.g., via `asyncio.all_tasks()` or `uvloop` metrics if available). Ensure throughput doesn't bottleneck on embedding.

5.  **Optimize `/collections` N+1 Query (High #2):**
    *   **Description:** `list_collections` performs an N+1 query pattern, with `distinct_doc_ids` scrolling every point in each collection.
    *   **Action:**
        *   Implement a bounded scroll for `distinct_doc_ids` (e.g., `limit=100`, only enough to get an approximate count for display).
        *   Add a short-lived in-memory cache (`cachetools.TTLCache(maxsize=64, ttl=5s)`) around `QdrantService.list_collections` and `QdrantService.get_collection_info` to prevent repeated expensive calls.
    *   **Validation:** Measure latency of `GET /collections` with many documents; ensure it remains fast (e.g., &lt;100ms) and doesn't scale linearly with total points.

### Phase B — Authentication, Rate Limiting, &amp; Request Correlation

**Goal:** Secure the API and improve debuggability with request tracing.

1.  **Implement Optional API Key Authentication (Critical #2):**
    *   **Description:** The API is completely open, posing a security and cost risk.
    *   **Action:**
        *   Add a new environment variable `RAG_API_KEY` to `app/config.py` (default to empty string).
        *   Create a FastAPI dependency `verify_api_key` that checks `X-API-Key` header against `RAG_API_KEY` (if set). Use `secrets.compare_digest` for constant-time comparison.
        *   Apply this dependency to protected endpoints: `/ask`, `/compare`, `/documents/*`, `DELETE /collections/{name}`.
    *   **Validation:**
        *   With `RAG_API_KEY` unset, verify protected endpoints are accessible.
        *   With `RAG_API_KEY` set, verify protected endpoints require the correct `X-API-Key` and reject incorrect/missing keys with a 401/403.

2.  **Add Rate Limiting (Critical #2):**
    *   **Description:** No rate limiting, enabling abuse and cost overruns.
    *   **Action:** Integrate `slowapi` (or similar) to enforce per-IP rate limits on protected endpoints (`/ask`, `/compare`). Configure limits via environment variables (e.g., `RATE_LIMIT_ASK`, `RATE_LIMIT_COMPARE`).
    *   **Validation:** Test hitting `/ask` or `/compare` beyond the configured limit and verify a 429 Too Many Requests response.

3.  **Implement Request ID Correlation (High #4):**
    *   **Description:** `request_id` is generated ad-hoc by error handlers but not consistently generated, propagated, or logged.
    *   **Action:**
        *   Modify `app/middleware.py` to generate a `request_id` (honoring an inbound `X-Request-ID` header if present, else generating a new UUID) at the start of each request.
        *   Store the `request_id` on `request.state`.
        *   Add `request_id` to the logging context (e.g., via `logging.Filter` or `LogRecord.process`).
        *   Ensure all logs emitted during the request include this `request_id`.
        *   Ensure the `request_id` is consistently returned in all API responses (including success and error envelopes).
    *   **Validation:** Send a request (with and without `X-Request-ID` header); verify the `request_id` appears in server logs and all API responses.

### Phase C — Dependency Migrations &amp; Robustness

**Goal:** Update deprecated dependencies and improve startup validation.

1.  **Migrate `pypdf2` to `pypdf` (High #6):**
    *   **Description:** `pypdf2` is deprecated; `pypdf` is the maintained successor.
    *   **Action:**
        *   Update `requirements.txt`: replace `pypdf2==3.0.1` with `pypdf==X.Y.Z` (latest stable).
        *   In `app/services/parser.py`, update imports from `import PyPDF2` to `import pypdf` and adjust API calls (mostly `PdfReader` usage is similar).
    *   **Validation:** Run unit tests for `FileParser` (e.g., `test_parser.py`) and integration tests involving PDF uploads.

2.  **Rewrite `GeminiProvider` for `google-genai` (High #6):**
    *   **Description:** The `google-generativeai` SDK is deprecated; `google-genai` is the new async SDK. The `gemini-1.5-flash` model is also on a deprecation path.
    *   **Action:**
        *   Update `requirements.txt`: replace `google-generativeai==0.8.3` with `google-genai==X.Y.Z` (latest stable).
        *   Rewrite `app/services/llm/gemini.py` to use the new `google.genai.Client` and its async API for `complete` and `complete_stream`. This is a significant rewrite.
        *   Update `GeminiProvider.model_id` to `gemini-2.0-flash`.
    *   **Validation:** Run `test_llm_factory.py` (ensure `GeminiProvider` builds) and integration tests (`test_api_ask_compare.py`) involving Gemini. Manually test `/ask` and `/compare` with Gemini to ensure streaming and non-streaming work.

3.  **Add Startup Provider Key Validation (High #5):**
    *   **Description:** The app starts successfully even if LLM API keys are missing, leading to runtime failures.
    *   **Action:** In `app/main.py`'s `lifespan` event, after loading settings, iterate through `AVAILABLE_PROVIDERS`. If `settings.groq_api_key`, `settings.gemini_api_key`, or `settings.anthropic_api_key` are empty strings, log a warning (or raise a `RAGAPIException` to fail fast in non-dev environments).
    *   **Validation:** Start the app with a missing API key and verify a warning log appears (or startup fails, depending on chosen severity).

### Phase D — Frontend Integration &amp; Fixes

**Goal:** Make the frontend functional and connected to the new backend contract.

1.  **Update Frontend API Calls for `/api` Prefix:**
    *   **Description:** Frontend `api.ts` currently calls root paths; backend will move business endpoints under `/api`.
    *   **Action:** In `web/src/services/api.ts`, change `const BASE_URL = ''` to `const BASE_URL = '/api'` (or similar, depending on the final nginx proxy config). Update `fetch` calls to reflect the new prefix. Update the `index.html` to load from `/`.
    *   **Validation:** Run the frontend in dev mode (`npm run dev`) and verify it can successfully call all backend endpoints.

2.  **Fix `Ask.tsx` Non-Stream Bug (High #8):**
    *   **Description:** The `handleAsk` function in `web/src/pages/Ask.tsx` only implements the `if (stream)` branch; when `stream` is `false`, no API call is made. Also, `onKeyPress` is deprecated.
    *   **Action:** Implement the `else` branch in `handleAsk` to call `api.ask` (non-streaming) when `stream` is `false`. Replace `onKeyPress` with `onKeyDown` or `onKeyUp` as appropriate.
    *   **Validation:** Toggle streaming off in the UI and verify that asking a question still produces an answer.

3.  **Add CORS Middleware (Critical #3):**
    *   **Description:** The FastAPI backend lacks CORS middleware, preventing cross-origin calls from the separately deployed frontend.
    *   **Action:** In `app/main.py`, add `CORSMiddleware` with configurable origins (e.g., `settings.cors_origins` from `.env`, defaulting to `http://localhost:3000`).
    *   **Validation:** Deploy frontend and backend separately (e.g., `npm run build` then serve `dist` from `web/` locally, and API via `uvicorn`) and confirm API calls succeed from the browser.

4.  **Create Frontend Docker Service &amp; Nginx Config (Critical #3):**
    *   **Description:** The frontend is undocumented and not served by Docker Compose.
    *   **Action:**
        *   Create `Dockerfile.web` (multi-stage build for React app, then nginx to serve static files).
        *   Create `web/nginx.conf` with a reverse proxy rule: `location /api/ { proxy_pass http://api:8000/api/; }` and `location /health { proxy_pass http://api:8000/health; }`, serving static files for all other paths.
        *   Add a `web` service to `docker-compose.yml` that builds `Dockerfile.web`, mounts `web/nginx.conf`, and exposes port 80 (or 3000, consistent with Vite dev).
    *   **Validation:** `docker compose up --build -d`. Verify the frontend loads in the browser and can interact with the backend API.

5.  **Document Frontend in README (Documentation Gap):**
    *   **Description:** The `web/` frontend is completely absent from project documentation.
    *   **Action:** Update `README.md` to include a section on the frontend, explaining its purpose, how to run it (dev and Docker Compose), and how it interacts with the API. Update the architecture diagram if needed.
    *   **Validation:** Review `README.md` to ensure the frontend is clearly explained.

### Phase E — CI &amp; Observability

**Goal:** Establish automated testing and foundational observability.

1.  **Implement GitHub Actions CI (Critical #5):**
    *   **Description:** No automated testing pipeline exists.
    *   **Action:** Create `.github/workflows/ci.yml` to:
        *   Set up Python 3.11 and Node.js.
        *   Start a Qdrant service container.
        *   Run `pytest` (including integration tests that hit the real Qdrant).
        *   Run `npm ci && npm run build` for the frontend.
    *   **Validation:** Push a PR and verify the CI workflow runs successfully, showing green for both backend tests and frontend build.

2.  **Add Real Qdrant Integration Test (Testing Gap):**
    *   **Description:** Existing integration tests are entirely faked.
    *   **Action:** In `tests/test_integration.py`, modify existing tests or add a new test that explicitly:
        *   Starts the FastAPI app with the *real* Qdrant client (configured via env vars to point to the Qdrant service container in CI).
        *   Uploads a small document.
        *   Performs an `/ask` query.
        *   Asserts the `/readiness` endpoint reports `"status": "ready"`.
    *   **Validation:** Run `pytest` locally (against a running `docker compose up`) and in CI; ensure this test passes.

3.  **Implement JSON Structured Logging (High #7):**
    *   **Description:** Logging is plain text, making it harder to parse and analyze in production.
    *   **Action:** Integrate `python-json-logger` (or similar) to output logs in JSON format. Ensure `request_id`, `provider`, `latency_ms` (from middleware), and `status_code` are included in log records.
    *   **Validation:** Observe server logs (e.g., `docker compose logs api`); confirm they are in JSON format with relevant fields.

4.  **Reconcile README/ARCHITECTURE with Reality (Documentation Gap):**
    *   **Description:** Several claims in `README.md` and `ARCHITECTURE.md` (readiness behavior, error leakage, auth, frontend) are contradicted by the code or are now outdated.
    *   **Action:** Review and update all relevant sections of `README.md` and `ARCHITECTURE.md` to accurately reflect the implemented changes (e.g., `RAG_API_KEY` for auth, `CORSMiddleware`, `/api` prefix, corrected readiness behavior, frontend existence).
    *   **Validation:** Manual review of documentation against the deployed system.

### Phase F — Verification

**Goal:** Final confirmation that all changes are integrated and working as expected.

1.  **Run Full Test Suite:**
    *   **Action:** Execute `pytest tests/ -v --cov=app --cov-report=term-missing` locally.
    *   **Validation:** All tests pass.

2.  **Smoke Test Docker Compose Deployment:**
    *   **Action:** Run `docker compose down && docker compose up --build -d` from a clean state.
    *   **Validation:**
        *   Both `api` and `web` services start and report healthy.
        *   `curl http://localhost:8000/health` returns `200 ok`.
        *   `curl http://localhost:8000/readiness` returns `200 ready`.
        *   Access the frontend in a browser (`http://localhost:3000` or equivalent) and verify basic functionality: document upload, `/ask`, `/compare` (including streaming), `/evaluate` all work correctly.
        *   Verify API key protection by setting `RAG_API_KEY` in `.env` and testing requests with/without the `X-API-Key` header.
